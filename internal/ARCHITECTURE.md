# Zelx Compiler Architecture

## Overview
Zelx is a compiler that transforms `.zx` source code into compilable C code. The architecture uses a transpilation approach with a pipeline consisting of several main stages.

## Core Components

### 1. Frontend (Parser & Lexer)
- **Lexer (src/lex.l)**: Uses Flex for tokenizing `.zx` source code
- **Parser (src/parser.y)**: Uses Bison (Yacc) for parsing tokens into AST

### 2. Intermediate Representation (AST)
- **AST Nodes (src/codegen.h)**: Data structures representing code
  - `ValueNode`: For expressions and values
  - `AssignmentNode`: For variable assignments
  - `MultiAssignNode`: For multi-assignments
  - `StringList` & `ValueList`: For lists and arrays

### 3. Backend (Code Generator)
- **Code Generator (src/codegen.c)**: Transforms AST into C code
- **Runtime Helpers**: C runtime functions for runtime operations

### 4. Runtime System
- **Value Union Type**: Dynamic type system using union to store various types
- **Runtime Functions**: Helper functions for array, string, comparison operations, etc.

## Compilation Pipeline

```
Source .zx → Lexer → Parser → AST → Code Generator → C Code → GCC → Executable
```

### Stage 1: Lexical Analysis (src/lex.l)
- Input: Source file `.zx`
- Output: Stream of tokens
- Tools: Flex (Lexical Analyzer Generator)
- Token types: Identifiers, literals, operators, keywords

### Stage 2: Parsing (src/parser.y)
- Input: Token stream from lexer
- Output: Abstract Syntax Tree (AST)
- Tools: Bison (Parser Generator)
- Grammar: Context-free grammar for Zelx language

### Stage 3: Code Generation (src/codegen.c)
- Input: AST from parser
- Output: C source code (`output.c`)
- Process: Tree-walking with inline code generation
- Features: Runtime helper injection, type handling, safety checks

### Stage 4: Compilation (GCC)
- Input: C source code
- Output: Executable binary
- Tools: GCC with flags `-Wall -Wextra -g`

## Architectural Design

### Separation of Concerns
- **Frontend**: Focuses on parsing and syntax validation
- **Backend**: Focuses on code generation and optimization
- **Runtime**: Focuses on execution and type safety

### Layered Architecture
```
┌─────────────────────────────────┐
│   Application Layer (Zelx apps) │
├─────────────────────────────────┤
│   Runtime Layer (C helpers)     │
├─────────────────────────────────┤
│   Code Generator Layer         │
├─────────────────────────────────┤
│   Parser Layer (AST)            │
├─────────────────────────────────┤
│   Lexer Layer (Tokens)          │
├─────────────────────────────────┤
│   Source Layer (.zx files)      │
└─────────────────────────────────┘
```

### Single Pass Compilation
The compiler uses a single-pass approach:
- Parser calls codegen functions directly during reduction
- No separate intermediate representation after AST
- Code generation happens inline during parsing

## Type System

### Dynamic Typing
Zelx uses dynamic typing with type inference:
- Types determined at runtime
- Union type for storing various data types
- Automatic type coercion for certain operations

### Supported Types
- **Integer**: `int` (C int)
- **Float**: `float` (C double)
- **String**: `string` (C char*)
- **Boolean**: `bool` (C int, 0/1)
- **Array**: Dynamic arrays with heterogeneous elements
- **Object**: Placeholder for objects (not fully implemented)

## Memory Management

### Strategy
- Manual memory management with `malloc`/`free`
- String duplication with `strdup`
- Memory allocation for AST nodes
- Memory leaks in AST acceptable (freed on program exit)

### Safety
- Array bounds checking at runtime
- Type checking for binary operations
- Null pointer checks for string operations

## Error Handling

### Compile-time Errors
- Syntax errors generated by parser (Bison)
- Line number tracking with Flex `yylineno`
- Informative error messages

### Runtime Errors
- Array index out of bounds
- Type mismatch in operations
- Assertion failures with `must()`
- All runtime errors result in `exit(1)`

## Performance Considerations

### Strengths
- Single-pass compilation (fast)
- Minimal transpilation overhead
- Direct C code generation (efficient)

### Limitations
- No optimization passes
- Runtime type checking overhead
- String concatenation not optimal (multiple allocations)

## Dependencies

### Build Tools
- **Flex**: Lexical analyzer generator
- **Bison**: Parser generator
- **GCC**: C compiler

### Runtime Libraries
- **Standard C Library**: `stdio.h`, `stdlib.h`, `string.h`
- **Flex Library**: `-lfl` for lexer runtime

## Build System

### Primary: build.zx
Build automation script written in Zelx itself:
- Clean, build, test commands
- Uses `must()` for error handling
- Dynamic build pipeline

### Alternative: Makefile
Traditional Makefile for compatibility:
- Standard GCC build process
- Clean and test targets
- Dependency tracking
